Hudi 历史渊源

Hudi（Hadoop Upserts Deletes and Incrementals）诞生于 Uber，在 2016 年之前，Uber 的数据基础设施主要依赖
Apache Hive，存储在 HDFS 上，并通过 Apache Spark 和 Presto 进行分析。但是，随着业务增长，
Uber 在数据处理方面遇到了几个主要挑战：

  1. 数据延迟高：

    • 传统的 Hive 处理模式是 批处理，每天只能更新一次数据（T+1）。
    • 由于数据刷新慢，Uber 的实时数据分析能力较差，无法及时响应业务需求，如订单匹配、地图优化、司机调度等。

  2. 存储和计算成本高：

    • 全量数据重写（Overwriting Full Dataset）：每次 ETL 任务都要重新写入整个表，导致 存储空间浪费严重，同时大幅增加了计算成本。

  3. 查询性能低：

    • Hive 依赖于 高并发全表扫描，查询大表时 查询性能很差，影响了业务团队的使用体验。

  4. 无法高效支持增量数据：

    • Uber 的数据是流式产生的，例如订单、支付、GPS 位置等，但 HDFS/Hive 不支持高效的 Upsert（更新） 和 Delete（删除） 操作。

🔍 Uber 的主要需求

为了应对这些问题，Uber 需要一个 高效的数据湖存储，必须满足以下要求：

  • ✅ 支持 Upsert（更新）和 Delete（删除）
  • ✅ 支持增量 ETL（只处理增量数据，避免全量重写）
  • ✅ 支持低延迟的数据湖查询
  • ✅ 降低存储和计算成本
  • ✅ 兼容 Hadoop/Spark/Presto/Hive

基于这些需求，Uber 在 2016 年 开发了 Hudi（Hadoop Upserts Deletes and Incrementals），并在 2017 年正式开源。

🛠️ Hudi 的核心特点

Hudi 通过 增量写入（Incremental Processing） 和 查询优化 解决了 Uber 面临的主要挑战：

功能 Hive（传统方法） Hudi（优化方法）
数据更新 不支持（只能重写） 支持 Upsert/Delete
数据插入 批量写入（慢） 流式写入（低延迟）
查询方式 全表扫描（慢） 增量查询 + 索引加速
存储优化 无增量数据支持 基于 Parquet/Avro 的增量存储
数据延迟 小时级或天级 分钟级甚至秒级

Hudi 借鉴了数据库的理念，在数据湖中引入了类似 索引、事务、增量变更 的概念，从而实现了高效的数据存储和查询。

📈 Hudi 在 Uber 的应用

Uber 通过 Hudi 优化了整个数据湖架构，主要用于：

  • 实时订单数据分析
  • ETL 任务加速
  • 机器学习特征存储
  • 日志数据存储
  • 报表和 BI 分析

Hudi 让 Uber 能够：

  • 将数据延迟从 24 小时降低到 30 分钟 🚀
  • 提升查询速度 100 倍以上 ⚡

🌍 Hudi 影响力 & 发展

  • 2017 年，Hudi 在 Uber 内部正式投入生产，并在 GitHub 开源。
  • 2019 年，Apache Hudi 进入 Apache 孵化器，成为 Apache 官方项目。
  • 2020 年，Apache Hudi 毕业成为顶级项目。
  • 现在，Hudi 被广泛用于 Netflix、Alibaba、腾讯、美团、字节跳动等公司 的大数据存储。

🎯 结论

Hudi 是 Uber 为了解决大规模数据湖的更新和查询问题 而开发的，它的成功推动了整个 数据湖 + 流批一体化 方向的发展 。
现在，Hudi 已成为 Apache Iceberg、Delta Lake 等数据湖格式的重要竞争者，并被许多大厂用于数据存储优化。
